## Resume Mode

### プロジェクト概要

MUSINSA 渋谷ポップアップ向けに、LINE 起点の試着予約・運用システムを開発。約 7,000 名の来場者が、LINE 予約とスタッフ管理 UI を通じてシームレスに試着できる体験を提供。

### 役割・成果

**フルスタック開発 / アーキテクチャ**

- NestJS（バックエンド）、Vue 3 + Vite（フロント）でエンドツーエンド実装し、LINE Messaging API / LIFF を統合
- DDD を適用し責務分離、要求変更への追従性を確保
- ポーリング型プリンターサーバーを設計し、設営当日の要件変更（複数階対応）を即日デプロイ
- AWS（ECS, RDS, ECR, Route 53, S3, CloudFront）へデプロイし、CloudWatch + AWS Chatbot を Slack 連携した監視を構築

**プロマネ / チームリード**

- 管理画面・顧客向け UI・サーバの 4 名チームを統括し品質と統合をリード
- タイトなイベント期限に合わせ、品質とスピードのトレードオフを管理
- 直前の要件変更（複数階プリンター設置）に柔軟対応（1 日で反映）

**機能・UX**

- 店頭 QR から商品をカートに入れ、試着リクエストを送信
- スタッフ向けリアルタイム進捗（pending → picking → ready → completed）
- 準備完了時に LINE 通知で呼び出し
- プリンタ連携で紙の管理票を自動出力

### インパクト

- 24 日間で約 7,000 名を **無停止・重大事故ゼロ** で対応
- クライアント/スタッフから運用効率化で高評価
- 100% 稼働率を達成（ロギング/監視/クリーンコードで事前防御）
- 手作業紙運用を置き換え、高負荷の同時試着リクエストを捌く仕組みを構築

### Demonstrated Skills

- **Backend:** NestJS, DDD, Event-Driven アーキテクチャ
- **Frontend:** Vue 3, Vite
- **Integration:** LINE Messaging API, LIFF, プリンタサーバ（ポーリング）
- **Infrastructure:** AWS (ECS, RDS, ECR, Route 53, S3, CloudFront), CloudWatch, AWS Chatbot
- **DevOps:** 充実したロギングと Slack へのリアルタイム通知
- **Soft Skills:** チーム調整、要件整理、迅速な問題解決、品質管理

### 学び

- DDD とクリーンアーキテクチャが、認知負荷を下げ評価基準を揃えることを再確認
- 期限のある案件での品質 vs スピードのバランス感覚を強化
- 無停止運用には先手の監視と防御的コーディングが有効であると実証

### 改善余地

- 着手前にチーム全体で DDD の勉強会を行い、共通の設計原則を早期に合わせる
- アーキ判断を担うテックリードの役割を明確化

## Storytelling Mode

### 課題: 渋谷で紙のカオスをなくす

韓国発のファッションブランド MUSINSA が渋谷で大規模ポップアップを開催。小さな空間で数千件の試着依頼を紙なしで回す必要があった。フロアを歩き、スマホで選び、準備できたら通知。クリップボードも名前呼びもなし。猶予は 2 か月。

### 基盤づくり

4 名チームを束ね、管理画面・顧客 UI・サーバを分担。求められるのは「動く」だけでなく「負荷下で破綻しない」こと。DDD を共通言語として採用し、コードレビューが衝突でなく対話になるようにした。バックエンドは NestJS + TypeScript、フロントは Vue 3 + Vite、顧客接点は LINE、インフラは AWS で堅牢性を担保。

### 想定外: 設営前日のプリンター増設

「2 階と 3 階にプリンターを置きたい」。前日になっての要望。だがプリンターサーバをポーリングでシンプルに設計していたため、EventDispatchService で複数プリンターにジョブを振り、サーバを拡張・デプロイして即日対応。柔軟性とは「変更が雪崩にならない設計」だと実感した瞬間。

### 本番: 7,000 人、ダウンタイムゼロ

10/3 に開場。QR で商品を読み、リクエストが飛び、スタッフがピッキングし、紙の伝票が出て、LINE で呼び出す。24 日間、重大障害ゼロ。ログ・監視・通知を徹底し、エッジケースを潰した防御的設計が効いた。コードは「賢く」ではなく「潔く」書いたので、異常時もすぐに分かり、平常時は静かに動いた。

### 人と DDD

技術的成果（ポーリング設計、EventDispatchService、監視）は大きかったが、最も印象的だったのは構造が人を変えたこと。DDD の原則で揃えたことで、コミュニケーションもレビューもスムーズになり、信頼が生まれた。スピードと品質の綱引きでは、感情面の折り合いをつけることも学んだ。

### もしやり直すなら

開始時に DDD の共通理解を先に作る。アーキ判断者を明確にする。それ以外はコアは正しかった。

### 最も響いた瞬間

初日、渋谷の会場で、お客さんが当たり前のように QR をかざし、スタッフが滑らかに動くのを見たとき、「これは見えないテクノロジーだ」と感じた。気づかれないくらい当たり前に動くこと。それが信頼であり、目指したものだった。

イベントは終わっても、学びは残った。クリーンコードは完璧さではなくレジリエンス。共通言語は摩擦を減らす。監視はパニックを防ぐ。そして、世界が一夜で変わっても 1 日で直せる設計こそ価値がある。

## Technical Deep-Dive

### システム構成

**Customer Flow:**

1. 店内で商品を見つける
2. 近くの QR を読み込み、LIFF で UI を開く
3. 商品をカートに追加
4. 試着リクエストを送信
5. 準備完了を LINE 通知で受信
6. 試着エリアで受け取り

**Staff Flow:**

1. 管理画面でリクエスト通知を受信
2. 在庫位置付きで依頼内容を確認
3. ピッキングを実施
4. プリンタで管理票を出力
5. ステータスを Ready に更新
6. 自動で顧客へ LINE 通知
7. 来場した顧客へセットを引き渡し
8. 完了して次の依頼へ

**Technical Components:**

- **Frontend (Customer):** Vue 3 + Vite + LIFF
- **Frontend (Staff Admin):** Vue 3 + Vite のリアルタイム管理 UI
- **Backend:** NestJS + DDD、明確な境界づけとユースケース分離
- **Integration:** LINE Messaging API（通知）、LIFF（組み込み体験）
- **Printer System:** ポーリング型プリンターサーバ + EventDispatchService
- **Infrastructure:** AWS ECS / RDS / ECR / Route 53 / S3 + CloudFront
- **Monitoring:** CloudWatch, AWS Chatbot (Slack), アプリログ

### 主要技術判断

**1. DDD 採用**

- ユビキタス言語で認識ずれを低減
- Bounded Context: Reservation / Inventory / Notification / Printing
- ユースケースパターンでビジネスロジックを分離
- EventDispatchService でコンテキストを接続

**2. ポーリング型プリンター**

- プッシュでなくポーリングにすることでネットワーク要件を簡素化
- ステートレス設計で複数プリンター対応が容易
- ジョブ管理はアプリ側の単一ソースオブトゥルースに集約

**3. 防御的設計と監視**

- すべての境界で詳細ログ
- エラー率/レイテンシ/リソースの CloudWatch アラーム
- Slack へリアルタイム通知
- 非クリティカル機能は優先度を下げたフェイルセーフ
