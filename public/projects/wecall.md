# WeCall

**期間:** 2024 年 10 月 ~ 2025 年 2 月  
**組織:** GENEROSITY Inc.  
**役割:** フルスタックエンジニア（クライアントサイド再設計・実装）  
**タグ:** #rearchitecture #serverless #aws #nuxt #typescript #cost-optimization #ux-design #polymorphism

---

## プロジェクト概要

WeCall は LINE を起点に、個人情報取得・予約・チェックインを提供するサービス。クライアント側アーキテクチャを全面的に再設計し、モノリスかつ高コストな構成をサーバレスかつモジュール化した構成へ移行。コストと開発期間を大幅に削減しつつ UX を改善した。

---

## 課題

元システムは多方面で問題を抱えていた。

**インフラコスト危機:**

- ECS + RDS 構成で常時稼働
- 環境ごとに約 $100（本番/ステージング/開発）
- 月次コストが約 200 万円に達し、年間 200,000 USD 規模
- ビジネスモデル的に持続不能

**開発非効率:**

- コンポーネントが密結合で再利用性ゼロ
- 案件ごとに DB スキーマから UI まで毎回改修
- 状態管理がストアにモノリシックに集約
- シンプル案件でも 2 週間以上
- コンポーネントライブラリや標準化が皆無

**技術的負債:**

- 個人情報取得・予約・チェックインが密結合
- パッケージ売りしかできず柔軟性なし
- 顧客要件に応じた組み替えが不可能

---

## 解決策とアプローチ

3 つの原則で再設計・実装。

### 1. サーバレスファースト

- ECS + RDS から AWS Lambda + DynamoDB へ移行
- 常時稼働コストを排除
- 性能を維持しつつ大幅なコスト削減

### 2. 動的フォーム駆動設計

- フォーム定義・バリデーション・コンポーネント設定を DB に移し、データドリブンで描画
- 過去案件データをテンプレ化して再利用
- DB 設定を解釈して動的にコンポーネントを評価・描画するポリモーフィック機構を実装
- 表示（フロント）とビジネスロジック/バリデーション（バック）を分離

### 3. 機能モジュール化

- 個情取得・予約・チェックインを独立モジュールに分割
- ポリモーフィック評価で自由な組み合わせを実現
- 機能を単品/セットで販売可能に
- 責務分離に SOLID を適用

## 技術実装

### アーキテクチャ

- **Frontend:** Nuxt.js (TypeScript)
- **Backend:** AWS Lambda（サーバレス）
- **Database:** DynamoDB
- **Validation:** Zod（Nuxt shared でフロント/バック共有）
- **Authentication:** AWS Cognito + LINE Access Token（二重トークンで LIFF 干渉を回避）
- **Infrastructure:** CloudFront, Route53
- **Analytics:** Kinesis → Firehose → Athena（DynamoDB を補う分析基盤）
- **Automation:** EventBridge + Step Functions（リマインダー通知）

### 主要技術判断

**共通バリデーション:**

- Nuxt shared に Zod スキーマを置き、クライアント/サーバ双方で実行
- 型安全と整合性を担保

**DynamoDB + Athena:**

- DynamoDB の分析弱点を Kinesis → Firehose → Athena で補完
- SQL ベースの集計・BI を実現

**二重認証戦略:**

- Cognito と LINE アクセストークンを別系統で扱い、LIFF との干渉を回避（理想より実用を優先）

**ポリモーフィックコンポーネント:**

- DB に保存された設定を実行時に解釈して描画
- 表示はフロント、検証は共有 Zod で実施
- コアを改変せず新機能モジュールを追加可能

**分析・監視:**

- DynamoDB → Athena パイプライン
- ビジネスインサイトを得るクエリを提供

**自動化:**

- EventBridge によるリマインダーワークフロー
- Step Functions で多段通知ロジックを構築

## 課題と対応

### 課題1: DynamoDB の分析制約

**問題:** 分析クエリに不向き。
**対応:** Kinesis → Firehose → Athena パイプラインを構築し、運用 DB は DynamoDB のまま SQL 分析を実現。

### 課題2: 機能モジュール化

**問題:** 個情 + 予約 + チェックインの密結合で柔軟販売が不可。
**対応:** ポリモーフィズムと SOLID で責務分離し、DB 設定に基づいて独立モジュールを動的評価。

### 課題3: 期限の圧迫

**問題:** タイトなスケジュールと複雑要件。
**対応:** 外部有識者（10xdev）へ相談し、優先度を絞り、再利用可能なパターンで高速実装。

### 課題4: 認証の複雑性

**問題:** OIDC 実装が LIFF と衝突する恐れ。
**対応:** Cognito + LINE の二重トークン戦略を採用し、懸念を分離。理想より実運用を優先。

## Reflection（Storytelling）

この案件は技術的成長の転機だった。開始時は月 200 万円のコストと週単位の開発サイクル、変更を拒むコード——非効率の塊。だが同時に、大きな改善余地があると感じた。

「良いアーキテクチャはビジネスを変える」を証明したかった。締切は厳しく、DynamoDB も初経験、95% コスト削減と機能向上を同時達成するプレッシャー。特に DynamoDB の制約や LIFF との両立で理想と現実のバランスに悩んだ。

支えになったのは「再利用できる仕組みを作る」ビジョン。ポリモーフィズムと SOLID を腹落ちするまで考え、責務分離を徹底。10xdev に相談し、夜遅くまで結合を疑い続けた。

デプロイ後、$100→$5/環境、2 週間→5 日の数字を見たときの実感は大きかった。さらに UX 向上を顧客が認めてくれたことで、エンジニアリングとデザインの両面が効いたと分かった。

大切なのはパターンをなぞることではなく、課題を深く理解し、実用的で美しい解を作ること。曖昧さに耐え、学び続け、スケールする仕組みを届けられる自信を得た。もっと Open/Closed に寄せ、テスト性と拡張性を高めたい——そう思い続ける姿勢を次に活かしている。
