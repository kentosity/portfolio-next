## 📋 Resume Mode

### プロジェクト概要

LINE を入口とし、注文〜スタッフのピッキングまでをデジタル化したイベント特化の在庫・注文管理システム。ポップアップ会場で紙伝票のオペレーションを廃止し、リアルタイム通知で受け渡しを効率化。

### 担当

- **フルスタック開発:** スタッフ用/管理用ダッシュボードのフロント・バックを設計/実装
- **リアルタイム基盤:** Socket.io + Redis PubSub（ElastiCache）で分散サーバ間を同期する WebSocket アーキテクチャを構築
- **プリンタ連携:** Express + TypeScript でプリンタサーバを作り、自動印刷を実現
- **チーム支援:** コードレビューやデバッグでチームをサポートしつつ納期を順守

### 技術スタック

- **Frontend:** LIFF, Vite, Vue.js
- **Backend:** NestJS, Express（プリンタサーバ）
- **Database:** AWS RDS
- **Infrastructure:** AWS (ECS, ECR, Route 53, CloudFront, S3, ElastiCache, ALB)
- **Real-time:** Socket.io + Redis PubSub
- **Integration:** LINE Messaging API, Epson TM-T88VII

### 実装機能

**スタッフ操作ダッシュボード:**

- リアルタイム注文一覧とステータス管理
- 在庫トラッキング・調整
- ピッキング/フルフィルメント導線
- 顧客通知
- 注文票のプリント出力

**管理ダッシュボード:**

- 注文全体の分析とモニタリング
- 在庫管理・調整
- イベント設定・構成

### 成果

- YSL ポップアップで稼働（[PR Times](https://prtimes.jp/main/html/rd/p/000000308.000032072.html)）
- イベント期間中に数千件の注文を処理
- 紙運用をデジタル化し **待機列を解消**
- スタッフ/クライアントから CX 向上で好評価
- アーキテクチャは後続プロジェクト（WeFit）にも流用

### 技術的成果

- ElastiCache (Redis PubSub) を使い、複数 ECS インスタンスでリアルタイム更新を同期する分散 WebSocket 構成を実装
- 冗長構成を考慮したプリンタサーバを設計
- デザイン知識を活かし、UX の改善提案を実行
- 責務分離でクリーンなコードを維持し、短納期下でバグを最小化

### 課題と対応

**課題:** 初のチーム開発でメンバーのスキル差が大きい  
**対応:** 自分のタスクを早期完了し、レビュー/リファクタで支援

**課題:** 2 か月の短期で品質を保つ  
**対応:** 責務分離と読みやすさを最優先し、結果的に開発速度を向上

**課題:** インフラ知識が不足  
**対応:** ECS/ElastiCache などをハンズオンで習得し分散パターンを吸収

### 学び

- **コード品質の基礎:** バグの少ない保守性がチーム開発では最重要
- **チーム開発:** コードレビューと認知負荷配慮の実践を初体験
- **スケーラブル設計:** 分散・リアルタイム通信の実地経験
- **UX 駆動:** デザイン思考を実装に織り込み、体験を改善

## 🎨 Portfolio / Storytelling Mode

### はじめての「火の中」

入社直後、試用期間でいきなり大案件。それが WeCart。3 人チーム、2 か月、クライアントは YSL のポップアップ。自分のコードを何千人が使う——そのワクワクが勝っていた。

### 解いた課題

会場で紙の注文票を手に並ぶ、あの長い列をなくすこと。LINE で注文し、準備できたら通知。シンプルだが体験は劇的に変わる。並ばず、待たず、好きに過ごせる。

### 初めてのチーム開発

個人開発ばかりだった私にとって、バグの多さに驚いた。誰のせいでもなく、スキル差やコミュニケーションの課題が形になっただけだった。ここで芽生えた問いがある。

**「どうすればバグの少ないソフトウェアを書けるか？」**

この問いがキャリア全体の軸になった。

### 技術の旅

スタッフ UI と管理 UI を担当し、リアルタイム注文、在庫同期、分散 WebSocket、プリンタサーバまで実装。インフラは初学習だったが、ECS/ElastiCache/Route 53 などを吸収しながら進めた。複数 ECS を同じ WebSocket 空間で同期する解は Redis PubSub。Express + TypeScript で作ったプリンタサーバは冗長構成も考慮した。

### 時間との競争

延期のない 2 か月。速さの鍵は「速く書く」ことではなく **正しく書く** こと。責務分離と読みやすさでバグを抑え、結果的に速度が出た。自分のタスクを終えてからレビューに回り、時には書き直す。それが最短だった。

### コードレビューで変わった視点

「相手が見やすく、認知負荷を下げる」——この言葉が残った。コードはみんなで読むもの。未来の自分も読む。だから読みやすく書く。

### UX とエンジニアリング

デザイン知識を活かし、UX の悪い箇所に提案を入れた。エンジニアは仕様実装だけでなく、体験を良くする提案者になれる。

### ローンチ

イベント当日、長い列は消え、LINE で通知を受けた人がスムーズに受け取りに来る。スタッフもクライアントも「並ばなくて良かった」と評価。画面越しに誰かの体験が良くなる——それだけで報われた。

### レガシー

WeCart のアーキは後の WeFit に流用された。しかし一番の遺産は、

- チーム開発の難しさと楽しさ
- バグの少ないコードの重要性
- インフラ/スケールの理解
- UX と開発の融合
- 「なぜ良いソフトウェアを作りたいのか」という問い

ここが、エンジニアとしての旅の始まりだった。

## 📊 Technical Deep Dive

### システムアーキテクチャ

```
[User (LINE LIFF)]
    ↓
[CloudFront + S3 (Frontend)]
    ↓
[ALB] → [ECS (NestJS x Multiple Instances)]
    ↓
[ElastiCache (Redis PubSub)] ← WebSocket sync
    ↓
[RDS (PostgreSQL/MySQL)]

[ECS Instances] → [WebSocket] → [Printer Server (Express + TS)]
    ↓
[Epson TM-T88VII]
```

### ユーザーフロー

1. LINE から LIFF にアクセス
2. 商品閲覧・注文
3. バックエンドで注文を受け、在庫をリアルタイム更新
4. スタッフがダッシュボードで受注を確認
5. ピッキングしてステータス更新
6. 注文票を自動印刷
7. 準備完了を LINE で通知
8. 受取カウンターで商品を渡す

### リアルタイム通信

- Socket.io による WebSocket 接続
- Redis PubSub (ElastiCache) でサーバ間ブロードキャスト
- 全 ECS インスタンスで同一の WebSocket 状態を共有
- 注文/在庫のリアルタイム同期を全クライアントへ反映

### プリンタ連携

- サーマルプリンタと同一ネットワークに Express サーバを配置
- メインアプリと WebSocket 接続し印刷ジョブを受信
- ESC/POS に整形して Epson TM-T88VII に出力
- 複数プリンタサーバにも拡張しやすい冗長設計
