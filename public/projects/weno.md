## プロジェクト概要

WeNo はイベント会場向けに特化した、LINE ベースのチケット発券・呼び出しシステム。常設店向けの既存版とは異なり、数千人が同時に限られたチケットを取り合う高負荷環境に対応。2 段階の一括呼び出し通知やリアルタイムの隊列管理を LINE 上で完結させる。

**一言サマリ:** 数千枚が数秒で消える Spotify ポップアップでも耐えた、高負荷対応の LINE チケットシステム。

- **5 件以上のイベントを無停止で成功**、満足度が高くリピート獲得
- ECS/RDS/CloudFront/ElastiCache/ALB など AWS インフラをフル構築し、GitHub Actions で CI/CD

- **二段階通知:** T-10 分警告とチェックイン開始の 2 ステップ一括呼び出しを実装
- **DB 最適化:** 時間帯別リリースとチェックイン管理に対応するスキーマへ改修
- **インフラ自動化:** CloudWatch + AWS Chatbot で Slack 通知、ゼロダウンタイムデプロイを整備

## Technical Deep Dive

### アーキテクチャ

```
User (LINE App)
    ↓
LIFF Frontend (Vite + Vue)
    ↓ API (CloudFront)
Backend API (Next.js on ECS)
    ↓
BullMQ (Queue)
    ↓
RDS (PostgreSQL) + ElastiCache (Redis)
    ↓
LINE Messaging API (Notifications)
```

### 技術スタック

**Frontend:**

- Vite
- Vue.js
- LIFF（LINE Front-end Framework）
- S3 配信、CloudFront 経由

**Backend:**

- Next.js（API Routes）
- BullMQ（ジョブキュー）
- LINE Messaging API
- AWS ECS (Fargate) で稼働

**Infrastructure (AWS):**

- **Compute:** ECS (Fargate)、SAM でリリース前にオートスケール
- **DB:** RDS (PostgreSQL)
- **Cache:** ElastiCache (Redis) — BullMQ とセッション
- **CDN:** CloudFront（Vue/Next を振り分け）
- **ALB:** Application Load Balancer
- **DNS:** Route 53
- **Monitoring:** CloudWatch + AWS Chatbot → Slack
- **CI/CD:** GitHub Actions

**Key Libraries:**

- BullMQ（分散キュー）
- LINE SDK（メッセージ連携）

### 発券フロー

1. ユーザーが LIFF でチケットをリクエスト
2. CloudFront 経由で Next.js API へ
3. BullMQ にジョブを投入（Redis/ElastiCache 保持）
4. ワーカーが直列で処理しレースを排除
5. 空き枠があれば RDS に発券を書き込み
6. フロントからショートポーリングで完了確認
7. 完了後に LINE で通知

### 二段階一括呼び出し

**第 1 段（T-10 分通知）**

- 管理者が時間帯を選択
- 対象チケットを抽出し一括で LINE 通知
- 「10 分後に開始、会場へ向かってください」を送信

**第 2 段（チェックイン開始）**

- チェックイン開始時に最終通知
- LIFF からチェックイン可能
- 管理画面でリアルタイム進捗を表示

### 主要技術判断

**BullMQ を選んだ理由**

- 分散 ECS で従来ロックが効かず、他手段は失敗
- BullMQ が発券を直列化しレースを解消
- Redis ベースで exactly-once に近い処理を実現

**全面リライトを選択**

- 常設版と要件が乖離（時間帯リリース、チェックイン、多段通知）
- フロント UX も全く別物
- ビジネスロジックだけ残し、他を全リライトした方が速いと判断

**SAM で事前スケール**

- リリース時刻が予測可能なので、15 分前に ECS を増やす
- 負荷ピーク後は縮小しコスト最適化

## Race Condition を潰す

### 問題

多数同時リクエストで同じチケットが二重発券されるレースが発生（分散 ECS 間での整合不一致）。

```
User A が #100 を要求 → ECS1 で空き確認 ✓
User B が #100 を要求 → ECS2 で空き確認 ✓
両方が #100 を書き込み → 二重発券 ❌
```

Spotify のように初秒で殺到するイベントでは致命的。

### 失敗例

- DB ロック: 分散環境で効かず
- 楽観ロック: 高負荷でコンフリクト多発
- アプリ内セマフォ: インスタンス再起動で揮発

**誰も解決できていなかった。**

### 解決: BullMQ による直列化

```javascript
// 簡略イメージ
async function issueTicket(userId, timeWindowId) {
  const job = await ticketQueue.add('issue-ticket', {
    userId,
    timeWindowId,
    timestamp: Date.now()
  })

  return job.waitUntilFinished()
}
```

**効いた理由:**

- Redis を単一のソースオブトゥルースとして直列処理
- 分散 ECS 間でもジョブは一度に 1 件
- リトライ/失敗ハンドリングが標準装備

**トレードオフ:**

- 直列化でレイテンシは微増
- ただし「正しさ > 速度」。ユーザーは体感せず、秒間数千リクエストも処理できた

### 実証: Spotify イベント

数千人が同時に叩いた本番で **重複発券ゼロ**。システムは安定して動作。

## Challenges & Solutions

### Challenge 1: 極端な同時接続

**Problem:** チケット解禁と同時に数千人が殺到。

**Solution:**

- リリース 15 分前に SAM で ECS を自動スケール
- ElastiCache を事前ウォーム
- 適切なインデックスでクエリ最適化
- 結果: ダウンタイムゼロで処理

### Challenge 2: 「解けない」レースコンディション

**Problem:** 分散発券で二重発券。先人は解決できず。

**Solution:**

- 分散システムのパターンを調査
- BullMQ を選定し、キューを中心に再設計
- レースを完全排除

### Challenge 3: 短期での全面リライト

**Problem:** 既存コードを流用すると間に合わない。

**Decision:** フロントは全リライト、バックは業務ロジックを残しつつ大幅改修。

**Execution:**

- 約 1 か月で完了
- コードレビューで品質を維持
- 漸進改修より短期で仕上げられた

## What I Learned

### 技術の学び

**1. 分散環境のレースは本当に難しい**

- 従来ロックは分散で効かない
- 並列と戦うのではなく、キューで直列化する方が有効な場合がある
- 適切なツール（BullMQ）が「不可能」を平凡にする

**2. 速いが壊れるコードより、正しく動くクリーンコード**

- リライトは賛否あったが正解だった
- クリーンな設計で書き直す方が、レガシー修正より速いこともある
- コード品質は開発速度に直結する

**3. オートスケール戦略が鍵**

- 予測できる負荷には事前スケールが有効（SAM）
- 事前ウォームでコールドスタートを回避
- イベント後はスケールダウンしてコスト最適化

### プロダクトの学び

**1. ユーザーが求めるのは「正しさ」**

- 遅くても正しいチケットシステムが、速いが壊れるものより価値がある
- BullMQ のレイテンシを気にする人はいないが、重複発券は致命的

**2. 本番以上のテストは存在しない**

- 負荷試験は役立つが、Spotify ファンの本番トラフィックに勝る教材はない
- 実戦で分散システムを学んだ

### Personal Growth

**1. 自分の判断を信じる**

- BullMQ を提案したとき、自分でも正解か確信はなかった
- それでも調査・検証し、解決策としてコミットした
- 誰も解けていなかった課題を、自分が解く覚悟を持った

**2. クリーンアーキテクチャは効率を上げる**

- リライトで、良い設計から書き始める方が速いと実感
- 以後の全プロジェクトで「まず設計を正しく」が指針になった

**3. プレッシャー耐性**

- 数千人の実ユーザーが依存する緊張感は大きい
- それでも落ちずに動いた瞬間の満足感は大きい

## Reflections & Storytelling

### The Impossible Bug

参画時、負荷がかかるとチケットが重複する「幽霊バグ」があった。ロックやトランザクション、楽観ロックも試されたが解決せず。

気づいたのは、**戦う場所を誤っていた** こと。システムは複数の ECS 上に分散しており、従来ロックはマシンをまたぐと効かない。

### 正しいツール探し

週末を使って分散システムのパターンを調査。Redis ロック、分散トランザクション、メッセージキュー……そこで BullMQ に出会った。

発想はシンプル。「各インスタンスが並列で奪い合う」のをやめ、**列に並ばせる**。全リクエストをキューに入れ、順番に 1 件ずつ処理する。

実装して負荷試験をすると、重複が完全に消えた。

### 数千人が襲来した日

初回の Spotify イベントは正直怖かった。カウントダウンがゼロになった瞬間、メトリクスのグラフが垂直に跳ね上がり、初秒で数千リクエスト。

それでもシステムは耐え、1 秒未満で完売しつつも重複もクラッシュもゼロ。ほっと息を吐いたのを覚えている。

### リライトが教えてくれたこと

「時間がかかる」と反対もあったが、貼り合わせだらけの既存コードを直すより、設計から書き直す方が速いと判断。フロントはゼロから、バックは業務ロジックを残しつつ組み直し、1 か月で完遂。

結果、クリーンスタートの方が速かった。レガシーバグやワークアラウンドと戦うより、設計が正しい方が開発は速い。**最速の道が「作り直し」になることもある** と学んだ。

### 誇りに思うこと

一番誇らしいのはインフラやオートスケールではなく、熱量の高いファンが争奪戦をしても **システムが裏切らなかった** こと。エンジニアリングの価値はそこにある。

### もしやり直せるなら

正直、時間に余裕があれば DB レベルで解く。PostgreSQL の `SELECT FOR UPDATE SKIP LOCKED` とリトライで直列化せずに処理できたはず。

BullMQ は時間制約下での最適解だったが、キュー・Redis・ワーカー管理という複雑さを増やした。理想解は:

- `SELECT FOR UPDATE` で行ロック
- エクスポネンシャルバックオフ付きリトライ
- 適切なインデックスとクエリ最適化

とはいえ当時の自分は DB 並行性への理解が不足していた。知っているツールで解き、期限に間に合わせた。**常によりクリーンな解はあるが、動くものを期限内に出すことが最優先** という教訓。
