## レジュメ対応サマリー

### プロジェクト概要

LIFF（LINE Front-end Framework）とGoogle Apps Scriptを使用してLINEプラットフォーム上に完全無料の個人財務追跡チャットボットを開発。日常的に使用されているメッセージングインターフェースを通じて、摩擦のない支出ロギングを実現した。

### 主な成果

- **継続的な使用**: 連続12ヶ月間、毎日のエントリで個人財務を正常に追跡
- **ゼロインフラストラクチャコスト**: 無料層のサービス（GAS、Google Sheets、LINE Messaging API）を使用したアーキテクチャ
- **豊富な可視化**: 支出を色の濃淡で表示するカレンダーヒートマップと、支出と収入を追跡するプログレスバーを実装
- **完全なデータパイプライン**: LIFFフォーム入力→ウェブフック処理→スプレッドシート保存→フロントエンド集計と可視化までの一連のシステムを構築

### 技術的実装

- **フロントエンド**: LIFF（LINE Front-end Framework）とJavaScriptを使用したフォーム入力とクライアント側のデータ集計
- **バックエンド**: ウェブフックエンドポイントとAPIレイヤーとして機能するGoogle Apps Script
- **データベース**: 構造化データ保存およびダイレクト編集機能を備えたGoogle Sheets
- **統合**: LINE Messaging APIを使用したボット対話とリプライメッセージ
- **データ形式**: 6フィールドトランザクションモデル（日付、名前、価格、メモ、カテゴリ、通貨）

### 解決した問題

既に日常的に使用しているメッセージングアプリ（LINE）を活用することで家計管理の摩擦を排除し、既存アプリのサブスクリプション料金を回避しながら、無料層では利用できない（日別・月別・カテゴリ別の内訳）カスタマイズ可能なビューを作成した。

### スキル実証

- API統合（LINE Messaging API、LIFF SDK）
- フルスタック開発（フロントエンドフォーム設計、バックエンドウェブフック処理、データベース操作）
- 自己主導的学習（LINEプラットフォーム初体験、ドキュメント通じた学習）
- 制約環境でのイテレーティブなデバッグ（GASロギング、頻繁なデプロイメント）
- プロダクト思考（既存ソリューションの摩擦点を特定、個人ワークフロー向けに設計）

2020年、私は大学生で、Introduction Inc.でインターンをしていた。プロの開発者として初めてコードを書き始めた時期だった。JavaScriptはまだ新しく、毎回のコミットが小さな勝利に感じられた。しかし仕事以外で私を悩ませていた問題があった。それは自分のお金がどこへ行っているのか全く分からないということだった。

多くの学生と同じように、私は限られた予算で生活していた。毎日のコーヒー、電車賃、コンビニでの買い物—すべてが積み重なっていたが、パターンが見えなかった。既存の家計管理アプリを試してみたが、すべてが同じ問題を抱えていた：

- 無料版は制限が多すぎ（カテゴリ別の内訳なし、月別集計なし）
- プレミアム版は持っていないお金がかかる

私が欲しかったのはシンプルなもの：フローを壊さずに支出をすばやくロギングし、自分のお金がどこへ行ったかを確認し、どれだけの収入を既に使用したかを知ること。

### チャレンジ：自分が実際に使用するもの構築

要件は私の心の中で明確だった：**LINEで支出を追跡する必要がある**。LINEが技術的に優れていたからではなく、私が既に常にそれを使用していたから。毎日、何十回も。支出をメッセージを送るのと同じくらい簡単にロギングできたら、もしかしたら—実際に続けられるかもしれない。

- LINE Messaging API

そして重要なことに、**予算がゼロだった**。大学生はHerokuのdynoやFirebaseのプランにお金を払わない。すべてが無料層で実行される必要があった。

しかし実際のチャレンジはアーキテクチャだった：LIFFフォームからのデータをどうやってウェブフックを通してスプレッドシートに入れ、また有用な形式で出すのか？

## 技術的な深掘り

### アーキテクチャ

```
ユーザー入力（LIFFフォーム）
    ↓
LINE Messaging API（ウェブフック）
    ↓
Google Apps Script（パーサー＋バリデータ）
    ↓
Google Sheets（データストレージ）
    ↓
Google Apps Script（APIエンドポイント）
    ↓
LIFFフロントエンド（集計＋可視化）
```

### データスキーマ

各トランザクションは行として保存：

- **日付**: トランザクション日付（YYYY-MM-DD）
- **名前**: ユーザー識別子
- **説明**: 何が購入されたか
- **価格**: 支出額
- **カテゴリ**: 支出カテゴリ（ユーザー定義）
- **通貨**: 通貨コード（例：JPY）
- **金額**: 正規化された金額

### 主要な設計決定

**LINEとLIFFを選んだ理由**

- ユーザーは既にLINEを常に開いていた
- アプリインストール不要
- 見慣れたインターフェースが摩擦を減らした

**Google Apps Scriptを選んだ理由**

- 完全無料（ホスティングコストなし）
- Google Sheetsとの直接統合
- 個人使用規模に十分
- 無制限に無料層で実行可能

**Google Sheetsを選んだ理由**

- 無料ストレージ
- 直接編集可能（UIを通さずにミスを修正）
- 見慣れたインターフェースでデータを確認
- エクスポート/バックアップが容易

**クライアント側集計を選んだ理由**

- シンプルなバックエンドロジック
- GAS実行クォータの問題が少ない
- 個人使用には許容可能なパフォーマンス（年間<1000トランザクション）

### 制限とトレードオフ

- **スケーラビリティ**: クライアント側集計は約10K行を超えてスケールしない
- **コード品質**: シングルファイルGAS実装、適切なアーキテクチャなし
- **手動入力**: 依然として手動のトランザクションロギングが必要
- **単一ユーザー**: 認証またはマルチユーザーサポートなし
- **オフライン不可**: 支出をロギングするためにインターネット接続が必要

### 今日だったら何が違ったか

- AIのOCR（GPT-4 Vision）を使用して領収書データを自動抽出
- キャッシング付きの適切なバックエンド集計を実装
- モダンスタック（Next.js + SupabaseまたはFirebase）を使用
- LLMを使用した自動カテゴリ分け
- 銀行/カード統合による自動トランザクションインポート
- 適切なエラーハンドリングと再試行ロジック

制約は明確だった：無料ツールとサービスだけを使用して機能的な家計管理者を構築すること。

## ソリューション

Luluは完全にLINE内に構築された家計管理者—日本で最も人気のあるメッセージングアプリ。ユーザーは友人とチャットするように対話でき、支出追跡を自然で簡単に感じさせる。

### なぜLINE？

1. **既にインストール済み:** 日本のみんなはLINEを使用
2. **見慣れたインターフェース**: チャットベースの対話は自然に感じる
3. **常にアクセス可能**: 移動中の支出をすばやくロギング
4. **無料プラットフォーム**: LINE Messaging APIには寛容な無料層

### なぜGoogle Apps Script？

1. **完全無料:** ホスティングコストなし
2. **サーバーレス:** インフラを管理する必要なし
3. **Google Sheets統合:** 無料データベース
4. **見慣れた言語:** JavaScriptベース

## 技術的チャレンジと解決策

### チャレンジ1：ゼロコスト要件

**問題:** ホスティングコストなしでバックエンドとデータベースが必要。

**解決:** Google Apps Script（サーバーレス、無料）とGoogle Sheets（無料ストレージ）を活用。この無料ツールの創造的な使用が、お金をかけずに機能するアプリを構築できることを証明した。

### チャレンジ2：自然言語パース

**問題:** ユーザーが特定のコマンド形式を学ぶ必要がない。

**解決:** ユーザーの意図を理解する柔軟なパターンマッチングを実装。「ランチ 850」も「850 for ランチ」も機能する。システムは寛容で、ユーザーの意図を理解しようとする。

### チャレンジ3：レスポンス時間

**問題:** Google Apps Scriptは遅い可能性、特に最初の実行時（コールドスタート）。

**解決:** スクリプト実行を最適化し、Sheets API呼び出しを最小化し、ユーザーデータのキャッシングを実装し、ユーザーの期待を設定。

### チャレンジ4：データプライバシー

**問題:** 金融データは機密、プライバシーを確保する必要。

**解決:** 各ユーザーが制限されたアクセス権を持つ独自のGoogle Sheetを取得。データはユーザー間で共有されない。金融情報の分析やトラッキングなし。

### チャレンジ5：信頼性

**問題:** 無料サービスはダウンタイムまたはレート制限の可能性。

**解決:** グレースフルなエラーハンドリング、有益なエラーメッセージ、再試行ロジックを実装。ステータスチェックとメンテナンスモード機能を追加。

## 学んだこと

### 技術的教訓

1. **制約駆動型イノベーション**
   - ゼロ予算の制約は創造的なソリューションにつながった
   - 時々制限は異なって考えるよう強制する
   - 無料は、巧妙に使用されれば劣っていない

2. **サーバーレスアーキテクチャ**
   - サーバーレスの最初の実体験（それが流行する前）
   - Function-as-a-Serviceパターンの理解
   - ステートレス実行向けの設計方法を学んだ

3. **チャットインターフェースデザイン**
   - 会話型UIは従来のUIとは異なる思考が必要

```javascript
function processMessage(message, userId) {
  message = message.toLowerCase().trim()

  // コマンドルーティング
  if (message === 'summary' || message === 'sum') {
    return generateSummary(userId)
  }

  if (message === 'chart' || message === 'graph') {
    return generateChart(userId)
  }

  if (message.startsWith('budget')) {
    return handleBudgetCommand(message, userId)
  }

  // デフォルト：支出としてパースしてみる
  const expense = parseExpense(message)
  if (expense) {
    recordExpense(userId, expense)
    return confirmationMessage(expense)
  }

  // 理解できなかった
  return helpMessage()
}
```

### スマートカテゴリ検出

```javascript
const categoryKeywords = {
  food: ['lunch', 'dinner', 'breakfast', 'meal', 'restaurant'],
  transport: ['train', 'bus', 'taxi', 'uber', 'gas'],
  entertainment: ['movie', 'game', 'concert', 'ticket'],
  shopping: ['amazon', 'mall', 'store', 'shopping']
  // ... より多くのカテゴリ
}

function detectCategory(text) {
  text = text.toLowerCase()

  for (const [category, keywords] of Object.entries(categoryKeywords)) {
    if (keywords.some((keyword) => text.includes(keyword))) {
      return category
    }
  }

  return 'other'
}
```

## 反省

Luluはリソースフルネスと創造的な問題解決を表します。ゼロ予算の制約に直面し、無料ツールを巧妙に組み合わせることでどのように有用なものを構築するかについて創造的に考える必要がありました。

プロジェクトは、最新フレームワークや高額なインフラストラクチャがなくても、有用なソフトウェアを構築できることを教えてくれました。時々、無料ツールを巧妙に組み合わせると、同等に効果的なものを作成できます。

さらに重要なことに、Luluは、個人的な問題を特定し、ソリューションを設計し、完全に自分で構築できることを自分に証明しました。その自信は私のキャリアで非常に貴重でした。

Luluはパーソナルプロジェクトで、プロダクトにはなりませんでしたが、重要なマインドセットを表しています：問題を見つけ、ソリューションを構築し、制約があなたを創造から止めるようにしないこと。
