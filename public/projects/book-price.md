# Book Price

**期間:** 2020 年  
**組織:** 個人プロジェクト  
**役割:** ソロ開発（フルスタック）  
**ステータス:** 完了（アーカイブ済み）  
**タグ:** #vue #javascript #api-integration #web-scraping #gas #automation #product-research

---

## プロジェクト概要

書籍のバーコードを読み取り、公開 API から商品情報を取得し、メルカリの出品をスクレイピングして中央値・平均値を算出、最適な出品価格の判断を支援する自動リサーチ Web アプリ。

**一言サマリ:** バーコードをスキャンするだけでメルカリの転売価格を即時分析する自動価格リサーチツール。

---

## 課題

### 問題

- メルカリ出品時に商品情報を手入力する必要があった
- 競合価格の調査は複数出品を手作業で検索・比較する必要があった
- 繰り返し作業で時間を取られ非効率だった
- 市場データに基づく最適価格を簡単に導き出せなかった

### なぜこのプロジェクトか

メルカリで本を売ろうとしていた際の個人的な不満から着想。日常的な面倒を解消するため、出品登録と価格リサーチを自動化することを目指した。

---

## 解決策とアプローチ

### コアコンセプト

スマホで使えるモバイルフレンドリーなインターフェースを用意し、以下を実現する:

1. スマホカメラで書籍バーコードをスキャン
2. 書籍データベースから商品情報を自動取得
3. 同一商品のメルカリ出品をスクレイピング
4. 出品価格の中央値・平均値を算出
5. すべてのデータをスプレッドシートに保存

### 技術戦略

- **フロントエンド優先:** Vue.js でレスポンシブかつモバイル最適な UI
- **サーバーレスデータ層:** GAS（Google Apps Script）でスプレッドシート連携
- **API 集約:** 複数の書籍 API を組み合わせて網羅的にデータ取得
- **Web スクレイピング:** Puppeteer でメルカリ価格をリアルタイム抽出
- **ローカル実行:** ホスティング不要にするためスクレイピングはローカル実行

---

## 仕組み

### ユーザーフロー

1. スマホで Web アプリを開く
2. カメラをバーコードに向ける
3. JavaScript ライブラリでバーコードを読み取る
4. 楽天ブックス API と Google Books API から商品情報を取得
5. Puppeteer で同一書籍のメルカリ出品をスクレイピング
6. 取得データから中央値・平均値を算出
7. タイトル・著者・ISBN・価格などをスプレッドシートへ自動反映
8. 集計データを確認して出品判断ができる

### 技術アーキテクチャ

- **フロントエンド:** Vue.js アプリ
- **バーコード読取:** カメラ対応の外部 JS ライブラリ
- **データ取得:**
  - 楽天ブックス書籍検索 API
  - Google Books API
- **価格スクレイピング:** ローカルで動かす Puppeteer でメルカリ抽出
- **データ保存:** GAS 経由で Google スプレッドシート
- **連携レイヤー:** GAS が API 呼び出しとシート書き込みを担当

## 課題と対応

### 課題: 初の公開 API 連携

- **Issue:** 公開 API 経験がなく手探りだった
- **Solution:** ドキュメントを読み込み、認証パターンを学び、エンドポイントを試行
- **Learning:** REST API の基礎（リクエスト/レスポンス、認証、エラーハンドリング）を習得

### 課題: 安定したバーコード読取

- **Issue:** モバイルカメラで精度の高い読取が必要
- **Solution:** モバイル最適化された外部 JS ライブラリを調査・導入
- **Outcome:** スマホカメラで安定した読取を実現

### 課題: メルカリデータ抽出

- **Issue:** 公式 API がなく価格データを直接取得できない
- **Solution:** Puppeteer によるスクレイピングで出品情報を抽出
- **Technical Detail:** デプロイやホスティングを避けるためローカル実行とした

## さらなる改善案

### いま作り直すなら

**スコープ拡張:**

- 出品まで自動化し、メルカリへ直接投稿
- マルチチャネル出品（メルカリ・ヤフオク・ラクマ等へ同時出品）
- 在庫同期を含むクロスプラットフォーム管理
- 自前の簡易 EC を選択肢として用意

**より高いマージン領域:**

- 書籍以外の高粗利カテゴリへ展開
- 家電・コレクティブル・ラグジュアリー商品に注力
- 典当（質屋）管理システムへの発展も視野
- 複数 SKU を一元管理できる在庫システム

**技術強化:**

- クラウド上での定期スクレイピング（ローカル依存の解消）
- 価格予測の機械学習
- 歴史的価格のトラッキングとトレンド分析
- 市場状況に応じた自動リプライス
- 画像認識による商品状態判定

**ユーザー体験:**

- まとめてスキャンするバルクモード
- 価格アラート通知
- 競合分析ダッシュボード
- 手数料込みの粗利計算機

## リフレクション / ストーリー

### きっかけ

メルカリで本を売るのが面倒、という単純なフラストレーションから始まった。ACE の経験直後で、身の回りの課題を技術で解消したかった。バーコードをスキャンすれば、出品に必要な情報と妥当な価格が一瞬で分かる——それが狙いだった。

### 学習の旅

2020 年は LLM もなく、解決策はドキュメントと Stack Overflow と試行錯誤だけ。初めて API 呼び出しが通り、実データが画面に表示されたときの高揚感はいまも覚えている。認証、JSON パース、エラーハンドリング——当時はすべてが新鮮なパズルだった。

メルカリに API がないため Puppeteer でサイトを自動巡回し、価格を集計したときは特に達成感があった。スクリプトが何十件も自動で集めて統計を出す様子は、まるで新しい力を手に入れたようだった。

### ピボットの瞬間

皮肉にも、このツールは「メルカリで本を売るな」と教えてくれた。手数料と送料を差し引くと利益がほぼ出ない。だが失望はなく、むしろ検証ができたことに納得した。作ったものが正しく答えを返しただけで、その答えが「やらない方がいい」だったという話だ。

### 何を意味したか

これは転売で稼ぐためではなく、課題を見つけて解決策を作れるかを証明するプロジェクトだった。API 連携、スクレイピング、データ分析を学び、コードが手作業を置き換える瞬間を体験した。

もっと重要だったのは、次のプロジェクトへの指針を得たことだ。実際の課題を解き、データで仮説を検証し、結果だけでなく学びを重視する——この姿勢はその後のすべてのプロジェクトで根幹になっている。

### 進化

振り返れば、これはより大きなシステムの種だった。マルチチャネル在庫管理、価格自動化、クロスプラットフォーム投稿——これらは単なる拡張ではなく、実ビジネスを支える要素だ。個人ツールとスケーラブルなプロダクトの違いは、スコープと実行の差に過ぎないと気づいた。

小さな個人プロジェクトにこそ、より大きなシステムの DNA が宿る——それが Book Price が教えてくれた最も価値のある学びかもしれない。
